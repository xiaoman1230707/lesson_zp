## 网络请求

- TCP 连接：三次握手建立连接。
    - 三次握手
    确认双方接受以及发送数据的能力
      - 建立连接
      - A 请求 B SEQ j 
      - B 响应 ACK j + 1 以及请求 A SEQ i
      - A 响应 B ACK i + 1
    
    - 四次挥手
    每一方的发送和接受要两次 握手四次
    第二次的时候可以响应对方的发送能力 同时希望对方确认我们的发送能力 
      - A 发送 FIN i 表示自己没有数据要发送了
      - B 响应 ACK i + 1 表示确认收到了
      - B 发送 FIN j 表示自己没有数据要发送了
      - A 响应 ACK j + 1 表示确认收到了
- TLS 握手：如果是 HTTPS，进行加密协商。
- 发送 HTTP 请求：等待服务器响应。
  - 发送请求行和请求头
        请求行： GET URL http版本
        请求头：jwt token Authorization: Bearer <JWT_Token> cookie 等
- 接收响应：处理重定向（301/302），接收 HTML 数据。
    头先于体 这允许客户端在接收完整个文件前就开始处理
    - 接收响应头和响应行 Headers
        响应行： HTTP/1.1 200 OK 等
        响应头：Response Headers (Content-Type, Content-Length, Set-Cookie...)
    - 头部预处理 决策点
        - 状态码 3xx? --> 读取 Location, 重新发起请求 (结束当前)
          http://time.geekbang.org 
          更加安全的协议 https 
          那么服务器会 返回301、302 重定向 服务器帮转法
          Location: https://time.geekbang.org
        - 状态码 304? --> 使用缓存 (结束当前)
        - 检查 CSP/HSTS 安全头 --> 违规则阻断
        - 检查 Content-Disposition --> attachment 则触发下载
    - 根据 Content-Type 分发
         |-- text/html -------> 交给 [ 渲染进程 ] 构建 DOM
         |-- text/css --------> 交给 [ 渲染进程 ] 构建 CSSOM (可能触发图片子请求)
         |-- application/json -> 交给 [ JS 引擎 ] 解析 (如果是 Fetch/XHR)
         |-- image/* ----------> 交给 [ 解码器 ] 解码像素
         |-- audio/video -------> 交给 [ 媒体播放器 ]
         |-- 其他/未知 --------> 触发 [ 文件下载 ]
    - 接收响应体 Body
        响应体：HTML 数据
- 通知：数据准备好后，通过 IPC (进程间通信) 通知浏览器进程。
  
🍽️ 餐厅类比：采购员（网络进程）去市场（互联网）买菜。先查哪家店有货（DNS），跟老板谈好价格建立关系（TCP/TLS），把食材（HTML 数据）买回来，告诉经理货到了。

## 提交导航阶段 (浏览器进程)

- 浏览器进程：接收到网络进程的数据。网络进程的报告
- 检查 Content-Type（确认是 HTML 还是下载文件）。
- 提交导航 (Commit Navigation)：更新地址栏 URL，更新会话历史（前进/后退按钮）。
  - 浏览器接收到"提交导航"的消息后，就检查beforeunload事件存在、入栈，执行页面卸载流程等，或者开启新页面，为之后渲染页面做准备。 
  - 最后渲染进程回想浏览器进程“确认提交”，已准备好接受和解析页面数据
  - 浏览器进程接收到渲染进程“提交文档” 消息后，便开始移除之前旧的文档，然后更新浏览器进程中的页面状态，进入loading状态
- 准备渲染进程：如果该域名有已有的渲染进程，则复用；否则创建新的 渲染进程。
- 建立通道：浏览器进程与渲染进程建立 IPC 通道，准备传输数据。

  
🍽️ 餐厅类比：经理（浏览器进程）确认买回来的是菜而不是其他东西。更新订单状态（地址栏），安排具体的厨师（渲染进程）接手，把食材交给厨师。


## 渲染阶段 (渲染进程)

    这是最复杂、最耗时的阶段，完全在 渲染进程 内部完成：

- 解析 (Parsing)：
- HTML 解析 -> 生成 DOM 树。
- CSS 解析 -> 生成 CSSOM 树。
- JS 执行：遇到 <script> 可能会阻塞解析（除非有 async/defer）。
- 构建渲染树 (Render Tree)：结合 DOM 和 CSSOM，剔除不显示的元素（如 display: none）。
- 布局 (Layout / Reflow)：计算每个节点在屏幕上的确切位置和大小。
- 分层 (Layer)：将页面分成不同的层（如视频层、滚动层），优化性能。
- 绘制 (Paint)：在每个层上绘制文字、颜色、边框等，生成位图。
- 光栅化 (Raster)：将位图转换为 GPU 能识别的网格（Tile），通常生成位图缓存。

🍽️ 餐厅类比：厨师（渲染进程）开始做菜。

洗菜切菜（解析 HTML/CSS）。
按食谱搭配（构建渲染树）。
决定摆盘位置（布局）。
把菜分到不同的盘子里（分层）。
炒菜装盘（绘制 & 光栅化）。
