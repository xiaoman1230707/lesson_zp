## 架构图

- 操作系统
  进程：分配资源的最小单位 process   老板
  线程：进程内的执行最小单位 thread  员工
  - 浏览器是多进程架构 整个过程需要各个进程之间的配合。
    - 浏览器主进程：
      - 主进程提供用户的操作反馈
        loaction 输入
      - 浏览历史 栈结构 入栈
      - 请求状态 loading 的显示
        网络请求开始 显示loading图标
        请求完成 隐藏loading图标
        请求的资源交给渲染进程 
      - 管理子进程
        进程间的通信 ipc Inter-Process Communication 
      - 当前已显示或者重新刷新
        beforeunload 事件 页面即将卸载（关闭、刷新或跳转）之前被触发
      浏览器主进程 负责用户交互，子进程 管理和文件存储(缓存、cookie、localStorage)
    - 网络进程 是面向渲染进程 和 浏览器进程等 提供网络下载功能
1. 首先，浏览器进程接收到用户输入的URL 请求，浏览器进程将该URL转发给网络进程
   - URL 帮助自动补全标准URL 带上http(s)://www.
   - 如果是搜索关键词，去向默认搜索引擎 URL带上关键词
   - 网址 英文、中文 前者可读性好 DNS
        OSI 七层协议 客户端 和 服务器 之间通过IP地址来通信的
        拿着域名(domain) 去查询IP地址
        像是 key:value 
        而DNS是一个分布式的数据库
        - 会先在本地DNS 缓存中查找
        - 局域网查找
        - 抚州电信 DNS服务器 查找
        - 中国根服务器 (海底光缆)
        - 美国根服务器 (顶级服务器)
    - 如果本地有缓存，
        text/css image/jpeg 
        不用请求，直接使用缓存
    - 三次握手 确保双方都具有请求 发送 和 接受 等能力
    - 发送请求行和请求头
        请求行： GET URL http版本
        请求头：jwt token Authorization cookie
    - 先收到的是 响应头 响应体慢慢发送。解析响应头数据，并将数据(头数据)转发给浏览器进程(做准备)。
        根据 Content-Type 浏览器会做出不一样的响应
        text/html -> 渲染进程准备接受
        text/css -> image下载
        doc MP3 ...
        状态码
    - 浏览器接收到"提交导航"的消息后，就检查beforeunload事件存在、入栈，执行页面卸载流程等，或者开启新页面，为之后渲染页面做准备。 
      再开始准备接受HTML数据，接受数据的方式是直接和网络进程建立数据管道。
    - 最后渲染进程回想浏览器进程“确认提交”，已准备好接受和解析页面数据
    - 浏览器进程接收到渲染进程“提交文档” 消息后，便开始移除之前旧的文档，然后更新浏览器进程中的页面状态，进入loading状态

    用户发出 URL 请求 到页面开始解析的整个过程 就叫做导航
