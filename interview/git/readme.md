# Git

## git pull 和 git fetch 的区别
- 分支 branch 
  独立开发时 一个人 一个默认主分支 只是做了版本的管理 但是也应该做分支的管理
  master/main 主分支 线上在运行的代码，正确的 不能乱改 尊重
  dev 开发分支  开发新功能 
  当如果多人协作时 每个人都会有自己的分支  
  可能同一个文件多人开发 分支不会相互影响 就能避免冲突
  切换到相应的分支 合并内容 
  改bug 很紧急
  git checkout -b bugfix 创建并切换 从主分支创建
  git merge branch_name 合并目标分支  合并后两个分支依然独立存在，名称也都在，除非删除分支。但是更改依然会保留
  feature 分支 
  最后还可以删除分支 不必管理那么多  
  git diff 查看差异 
  合并 推送 清理 远程清理

### git fetch
- 只拉取 ，安全又不影响当前分支，远程的main,拉取到本地的 origin/main下
- git diff main origin/main 查看差异
- git merge origin/main 
- 本地有自己的修改 不想立马合并
- git 自己的分支在开发，还没有完工(bug)
- 但是立马有个bug去修
- 修改多个文件，不方便立马去提交到仓库
  仓库：这是你的“档案柜”，每个分支都是一个独立的抽屉，里面存放着已经提交的代码快照。
  工作区：这是你的“办公桌面”。
  未提交的 修改是不属于任何分支的。
- dev 的修改不想带过去，也怕有冲突，在退出当前分支前存档
  当你修改了文件但没 commit 时，这些改动只是摆在办公桌面上。当你切换分支（git checkout）时，Git 会带着你的桌面一起挪到另一个抽屉面前。只要你的改动和新分支里的文件没有冲突，Git 就会允许你把这些“未竟的事业”带过去。
- git stash 保存当前分支的修改，切换到其他分支时，可以继续开发
  相当于把桌上的东西临时塞进一个备用抽屉，让桌面变干净。
- git stash pop 恢复当前分支的修改
  相当于把备用抽屉里的东西再放回桌面，继续开发。
- 切分支前必 commit 或 stash

### git pull
- 拉取远程更新，并自动merge到当前分支 

### git merge 和 git rebase
- merge 合并 缺点 会让提交历史变复杂，后期排查问题会变得很乱
- rebase 变基 优点 改变提交的基点，提交历史编程一条干净的时间线
