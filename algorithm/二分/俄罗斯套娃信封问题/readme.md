354. 俄罗斯套娃信封问题
已解答
困难

给你一个二维整数数组 envelopes ，其中 envelopes[i] = [wi, hi] ，表示第 i 个信封的宽度和高度。

当另一个信封的宽度和高度都比这个信封大的时候，这个信封就可以放进另一个信封里，如同俄罗斯套娃一样。

请计算 最多能有多少个 信封能组成一组“俄罗斯套娃”信封（即可以把一个信封放到另一个信封里面）。

注意：不允许旋转信封。

 
示例 1：

输入：envelopes = [[5,4],[6,4],[6,7],[2,3]]
输出：3
解释：最多信封的个数为 3, 组合为: [2,3] => [5,4] => [6,7]。
示例 2：

输入：envelopes = [[1,1],[1,1],[1,1]]
输出：1

###
错误做法：
宽升序，高升序
last = 0;
for (i from 1 to end)
while 当前信封不能放进 last → i++
找到了 → res++, last = i

但这是一种 贪心策略，只尝试一条路径，而实际需要找的是 最长嵌套序列，可能是非连续、非首尾相连的。

⚠️ 反例说明：
输入：

[[1,1], [2,2], [3,3], [4,4], [1,5]]
排序后（按宽升、高升）：

[1,1], [1,5], [2,2], [3,3], [4,4]
你的算法执行过程：

last = 0 → [1,1]
i=1: [1,5] 宽度相同 → 跳过 → i=2
i=2: [2,2] 满足 1<2 && 1<2 → 可以嵌 → res=2, last=2
i=3: [3,3] → 满足 → res=3, last=3
i=4: [4,4] → 满足 → res=4, last=4
最终结果：4

看起来对？但是看这条链：

[1,5] → [2,2]? NO! 因为 5 >= 2，不能嵌！
但问题是：你从 [1,1] 开始，确实能得到 [1,1]→[2,2]→[3,3]→[4,4]，共 4 层。

可如果改成：

[[1,1], [2,100], [3,3], [4,4]]
最优应是：[1,1] → [3,3] → [4,4]（长度 3）

但你的算法会选 [2,100]，然后 [3,3] 因为 100 > 3 被跳过 → 后面再也接不上！

→ 所以你一旦选了一个“高个子”，就会卡住后续更优路径。

🔴 结论：贪心跳跃不是动态规划或 LIS，不能保证全局最优！



在这个LIS算法里，为什么不一定为真的子序列，他的长度也和真正的最长升序子序列一样

核心思想：tails 是“状态表”，不是“路径记录”
tails[i] 的含义是：

“截至目前，是否存在一个长度为 i+1 的递增子序列，其最后一个元素 ≤ tails[i]？并且我们努力让它尽可能小。”

它是一个贪心状态压缩：

我们不在乎它是怎么来的
只在乎：如果将来有一个数字 x，只要 x > tails[i]，就能接上去，形成更长序列
💡 类比理解：打游戏升级
想象你在闯关：

每一关有一个“最小通关条件”：比如攻击力要大于某值
tails[i] 就是你打到第 i+1 关时，当前最弱但能通关的角色属性
你不需要记住你是怎么打过来的
只需要知道：“我现在可以用攻击力为 X 的角色挑战下一关”
即使这个角色是用后面捡的装备强化的，也不影响他能打下一关。

👉 所以 tails 就像你的“最强轻量级战士”集合，用来测试能不能继续升级。

✅ 为什么长度不会错？
因为算法满足两个重要性质：
✅ 性质 1：不会高估长度
只有当确实能找到一个比所有现有尾部都大的数时，才会扩展长度
每次扩展 tails.push(h) 都意味着找到了一条更长的递增链
所以 tails.length 是实际可达的最大长度的下界
✅ 性质 2：不会低估长度
我们总是保留每个长度下的最小尾部
这样能让更多后续数字有机会接上去
所以不会因为过早选了一个大数而卡住发展
👉 综合起来：

它既不会凭空变出不存在的长序列，也不会漏掉任何可能的扩展机会。

因此，最终长度一定等于真实的 LIS 长度

🧠 数学角度：归纳法思想
我们可以这样想：

假设处理前 k 个元素后，tails 正确反映了所有可能的递增子序列的最优尾部状态。

当你加入第 k+1 个元素 h：

如果 h 很大，能接在最长序列后面 → 扩展长度 → 更新最大值
如果 h 中等，只能接在某些中间长度后面 → 替换那个长度的尾部 → 让未来更容易扩展
如果 h 很小 → 改写长度为1的记录 → 为将来提供更多可能性
✅ 所以每一步都保持“状态最优”，最终结果自然正确。





为什么“高度降序”能防止同宽多选？
关键就在于：

在排序后，所有相同宽度的信封会连续出现，并且它们的高度是从大到小排列的

所以当你在后面求“高度的最长递增子序列”时：

你想从 [3,7] → [3,5]？不行！因为 7 > 5，不是递增
想从 [3,5] → [3,4]？也不行！5 > 4
所以你不可能在一个宽度内选出两个信封进入 LIS
👉 因为 LIS 要求“严格递增”，而同宽组内部是“递减”的 → 最多只能选一个（第一个被保留在 tails 中的那个最小尾部）

这就自动实现了：“每种宽度只选一次”

🚀 类比理解：贪吃蛇升级系统
想象你是一个贪吃蛇，要吃掉尽可能多的盒子来变长。

规则：

每个盒子有宽和高
你必须从小盒子吃到大盒子（严格大于）
但你不能吃两个一样宽的盒子连续出现（会被卡住）
如果你把盒子按“宽升序、同宽时高降序”排好：

[2,10] → [2,8] → [2,5] → [3,1] → [3,4] → [4,2]
你会发现：

在宽度=2这一组里，你是先看到大的高（10），再看到小的（5）
当你构建 LIS 时，你会保留最小的可能结尾
但你只能从 2 这一组中“贡献一个值”给后续更宽的组
这样就自然形成了跨宽度的增长链，且不会重复使用同一宽度。
